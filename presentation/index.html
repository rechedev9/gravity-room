Gravity Room

Programas probados de fuerza e hipertrofia
con progresión automática

Buenos días. Soy Luis Reche y os presento Gravity Room, una aplicación web para seguir entrenamientos de fuerza con progresión automática, mi proyecto del ciclo de Desarrollo de Aplicaciones Web. Voy a explicarlo de abajo arriba: primero la idea, luego veréis un mapa del sistema, y a partir de ahí subiremos desde la infraestructura hasta la interfaz.
El origen

El problema

Seguir un programa de fuerza con progresión a mano es un caos:

  • Cada sesión decides si subes peso, repites o bajas
  • Los cálculos manuales son tediosos y propensos a errores
  • Las hojas de cálculo se rompen en cuanto el programa se complica

La solución

Gravity Room automatiza la progresión de forma determinista.

Según cumplas o no el objetivo del día, la app ajusta sola la siguiente sesión —subir, repetir o bajar— con la misma regla siempre y sin cálculos a mano.

Los programas de fuerza populares —GZCLP, 5/3/1, StrongLifts y muchos más— son de progresión: cada sesión decides si subes peso, repites o bajas según cómo fue la anterior. Calcular eso a mano es tedioso y propenso a errores; la gente lo lleva en hojas de cálculo que se rompen enseguida. Esto nació de mi propia frustración entrenando. Gravity Room automatiza esa lógica: tú registras y la app decide.
Qué me propuse

Objetivos

  • Implementar un motor de reglas de progresión reutilizable para un catálogo de programas (GZCLP, 5/3/1, StrongLifts…)
  • Construir una app web (PWA) rápida y usable desde el móvil
  • Diseñar una API propia con autenticación segura
  • Desplegar el proyecto en producción real — no solo en local
Cuatro objetivos: primero, un motor de reglas de progresión reutilizable que sirva a varios programas —GZCLP, 5/3/1, StrongLifts y más—; segundo, una web instalable como app en el móvil; tercero, una API propia con login seguro; y cuarto, y esto era lo ambicioso, ponerlo en producción de verdad.

La base · capa 1

Despliegue e infraestructura

En producción, no en local 🚀

  • Servidor propio (VPS en Hetzner), todo en Docker Compose
  • 4 contenedores: Caddy + API + PostgreSQL + Redis
  • CI/CD con GitHub Actions: construye imágenes, compila la web y la sube al servidor en cada cambio
  • HTTPS automático con Caddy, sin configuración manual de certificados
Empezamos por la base. No se queda en mi portátil: está desplegado en producción en un servidor propio, todo orquestado con Docker Compose: el proxy Caddy, la API, PostgreSQL y Redis. Tengo integración y despliegue continuo con GitHub Actions, que en cada cambio compila todo y lo sube al servidor. Caddy gestiona el HTTPS y la renovación de certificados automáticamente. Este punto me diferencia de casi cualquier otro proyecto del ciclo: no lo escondo.
La base · el punto de entrada

Dominio y DNS

  • Dominio gravityroom.app registrado en Namecheap (feb 2026–2028)
  • Privacidad WHOIS activada — minimización de datos (RGPD)
  • TLD .app está en la lista HSTS preload de Google: el navegador fuerza HTTPS por diseño, sin SSL stripping
  • DNS delegado a Hetzner (nameservers personalizados) → registro y hosting en un único panel
  • 3 registros A → IP del VPS: ápex, www (301 → ápex) y api
Panel de Namecheap · nameservers delegados a Hetzner
Lo más bajo de todo: el dominio. Compré gravityroom.app en Namecheap, con privacidad WHOIS para no exponer mis datos. Elegí el TLD .app a propósito: está en la lista HSTS preload de Google, así que los navegadores rechazan HTTP por defecto y se fuerza HTTPS antes incluso de la primera respuesta. En vez de usar el DNS de Namecheap, delegué los nameservers al DNS de Hetzner para tener registro y hosting en un único punto, y configuré tres registros A apuntando al VPS: el ápex, www (que Caddy redirige al ápex) y api. Si cambia la IP del servidor, toco un solo registro.
Capa 2 · los datos

Base de datos

  • PostgreSQL con el ORM Drizzle (todo tipado)
  • 10 tablas modelando usuarios, programas y resultados
  • Migraciones automáticas al arrancar la API
Subimos a los datos. Uso PostgreSQL con Drizzle, un ORM que mantiene todo tipado en TypeScript: son diez tablas. El modelo central es sencillo de seguir: un usuario tiene programas de entrenamiento, y cada programa registra los resultados de cada serie. Las migraciones se aplican automáticamente al arrancar la API.
Capa 3 · la API

API propia

  • ElysiaJS sobre Bun (runtime más rápido que Node)
  • ~31 endpoints REST
  • Login con JWT + rotación de refresh tokens
  • Entrar también con Google (OAuth)
  • Contrato OpenAPI que genera el cliente tipado de la web
POST /api/auth/google
GET  /api/catalog/
GET  /api/programs/
POST /api/programs/{id}/results
POST /api/programs/{id}/undo
La API la diseñé yo, con ElysiaJS sobre Bun, que es más rápido que Node. Son unos 31 endpoints. El login usa JWT con rotación de refresh tokens —tokens de corta duración que se renuevan de forma segura— y permite entrar con Google. Además expone su contrato en OpenAPI, y de ahí genero automáticamente el cliente tipado que consume la web: si cambio la API, el cliente se actualiza y el build avisa si algo no cuadra.
El puente backend ↔ frontend

Una sola fuente de verdad

❌ Lo fácil: copiar las reglas en el cliente y en el servidor → se desincronizan y aparecen bugs.

✅ Mi decisión: un paquete compartido con TypeScript + Zod que además valida los datos con el mismo esquema en cliente y servidor. 

// packages/domain
// (web y API)

export const WorkoutResult =
  z.enum(['completed', 'failed'])

// Reglas del método: aquí,
// no en cada cliente
export function nextSession(
  prev: SessionState
): SessionState {
  /* reglas del programa */
}
Esta es la decisión de la que estoy más orgulloso, y une el backend con el frontend. Lo fácil habría sido copiar las reglas en cada cliente, pero eso genera bugs en cuanto uno se actualiza y otro no. En su lugar usé un paquete compartido con TypeScript y Zod que, además de centralizar las reglas, valida los datos con el mismo esquema en el navegador y en el servidor. Es la diferencia entre un proyecto de juguete y uno mantenible.
Capa 4 · la interfaz

Frontend

  • React 19 + Vite + TanStack Router/Query
  • Estilos con Tailwind 4
  • PWA instalable en el móvil como una app
  • Usable en el gym con mala señal (cache inteligente)
Llegamos a la interfaz. El frontend está hecho con React 19, Vite y TanStack para routing y datos, con Tailwind para los estilos. Es una PWA: se instala en el móvil como una app y sigue siendo usable en el gimnasio aunque haya mala cobertura, gracias a una estrategia de cache. Aquí lo veis en el móvil.
Cómo lo construyo

Calidad y proceso

  • Tests en web, API y dominio
  • Pruebas end-to-end con Playwright
  • Lefthook: typecheck, lint y tests automáticos antes de cada commit/push
  • CI que bloquea el cambio si el cliente de la API queda desincronizado

Y no es solo que funcione: lo construyo con un flujo profesional. Hay tests en el frontend, la API y el dominio, pruebas end-to-end con Playwright que simulan al usuario, y Lefthook ejecuta typecheck, lint y tests automáticamente antes de cada commit. La CI incluso bloquea el cambio si el cliente generado de la API queda desincronizado, y hago pruebas de carga con k6. (Slide recortable si voy de tiempo.)

Cómo trabajo

IA con disciplina

Uso la IA como copiloto, pero las reglas las pongo yo: un CLAUDE.md en el repo fija convenciones, estilo y TDD.

Tipado estricto e2e

Dominio, API y web comparten tipos: un contrato que verifica la máquina. Si la IA desincroniza capas, no compila.

ESLint sin escapatorias

Prohíbe any, as, @ts-ignore y !: la IA no puede tapar un error, debe arreglar la causa.

TDD

Primero el test que falla, luego el código mínimo: fija el comportamiento antes que la solución.

Tipos + lint son revisores automáticos en cada commit y en CI: la IA pasa por el mismo listón que yo.

(Detalle ampliado en notas-defensa.md, slide 13.) Uso la IA como copiloto, pero las reglas las pongo yo. El tipado estricto de punta a punta y el ESLint que prohíbe any, as, @ts-ignore y la aserción ! cierran las escapatorias con las que un LLM haría que algo compile sin arreglar el fondo. TDD fija el comportamiento antes. Resultado: tipos y lint son revisores automáticos en cada commit y en CI; el código de la IA pasa por el mismo listón que el mío.
